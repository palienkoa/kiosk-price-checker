# Price Check Kiosk — локальний бекенд

Локальний веб-сервіс для прайс-чекер кіоска (Symbol/Zebra MK500): «пікнув штрихкод → побачив назву, ціну, залишок, картинку». Кіоск — це тонкий клієнт-браузер рівня IE6, який просто рендерить HTML, що ми йому віддаємо. Дані періодично синхронізуються з материнської 1С і зберігаються локально в SQLite, тож кіоск працює навіть коли VPN до 1С лежить.

Навчальний проєкт на Python (Flask + SQLite). Не вимагає реального заліза кіоска для розробки — тестується з браузера телефона/ноута.

---

## Архітектура

```
[ Материнська 1С ]            [ Локальна коробка (Pi / Docker) ]        [ Кіоск MK500 ]
  у провайдера                  на тій самій LAN, що й кіоск             браузер IE6/CE
  ┌──────────────┐   VPN        ┌────────────────────────────┐   LAN    ┌────────────┐
  │ HTTP-сервіс  │ ───pull───►  │ sync-воркер  →  SQLite      │ ◄──GET── │ скан → URL │
  │ JSON каталог │   кожні      │ Flask        →  HTML + img  │ ──HTML─► │ рендер     │
  │ + картинки   │   ~10 хв     └────────────────────────────┘          └────────────┘
  └──────────────┘
```

Коробка виконує **дві ролі одночасно** в одному застосунку:
1. **Sync-клієнт** — раз на N хвилин через VPN тягне каталог JSON-ом з 1С, робить upsert у SQLite, докачує нові картинки.
2. **Веб-сервер кіоска** — той самий процес віддає MK500 простий HTML, читаючи з SQLite.

VPN-клієнт живе **тільки на коробці**. Кіоск про VPN не знає й торкається лише локальної мережі.

---

## Ключові рішення (і чому)

- **SQLite, а не повноцінна СУБД чи 1С.** Дані read-only, набір крихітний, навантаження мінімальне. Один файл, нуль адміністрування.
- **Pull, а не push.** Коробка сама ходить у 1С за розкладом. 1С нічого не «штовхає» й не мусить знати про кіоски.
- **Картинки — файлами в папці**, у БД лише шлях. Докачуємо тільки нові/змінені (за `changed_at`), а не весь набір щоразу.
- **Залишок завжди трохи застарілий** — рівно на інтервал синхри. Для прайс-чекера норм. Часто краще показувати `є / мало / нема`, а не точну цифру, щоб не ловити скарги.
- **HTML під IE6 / QVGA 320×240.** Жодних сучасних JS-фреймворків, `fetch`, AJAX. Звичайний GET-перехід зі штрихкодом у query.

## Стек

- Python 3.11+
- Flask — веб-роути + рендер HTML
- `sqlite3` (стандартна бібліотека) — локальна БД
- `requests` — pull із 1С
- APScheduler — розклад синхри в тому ж процесі
- Docker + docker compose — запуск

## Модель даних

```sql
CREATE TABLE products (
    barcode      TEXT PRIMARY KEY,
    name         TEXT NOT NULL,
    price        REAL,
    qty          REAL,
    picture_path TEXT,          -- шлях у static/images або NULL
    changed_at   TEXT,          -- мітка з 1С, для інкрементальної синхри
    synced_at    TEXT           -- коли ми останній раз оновили рядок
);
```

## Контракт із материнською 1С

1С віддає рівно те, що потрібно кіоску — не весь свій АПІ. Два endpoint-и:

**`GET /catalog?since=<ISO-час>`** — каталог, опційно лише змінене після `since`:

```json
{
  "items": [
    {
      "barcode": "4820000000001",
      "name": "Кава мелена 250г",
      "price": 142.50,
      "qty": 17,
      "img": "4820000000001.jpg",
      "changed_at": "2026-06-19T10:00:00"
    }
  ]
}
```

**`GET /image/<barcode>`** — бінарь картинки (jpg/png).

Це писати на стороні 1С (вузький HTTP-сервіс). На етапі розробки його замінює `mock_1c.py`, тож Python-частину можна довести до кінця без реальної 1С.

## План по етапах

Кожен етап завершується чимось запускним — зручно комітити окремо.

- **0. Каркас.** repo, venv, `requirements.txt`, hello-world Flask. *Ціль: тулінг працює, сторінка відкривається.*
- **1. БД.** `schema.sql`, `db.py` (init / upsert / get_by_barcode), засіяти 2–3 фейкові товари вручну. *Ціль: можу покласти й дістати товар.*
- **2. Сторінка кіоска.** `index.html` (поле вводу) + `product.html` (результат), читає з SQLite, GET-навігація `?barcode=...`. *Ціль: «ввів баркод → бачу ціну», перевірено з телефона.*
- **3. Mock 1С.** `mock_1c.py` віддає фейковий каталог JSON. *Ціль: є джерело для синхри без реальної 1С.*
- **4. Sync-воркер.** `sync.py` тягне з mock, upsert у SQLite, інкрементально за `changed_at`. Запуск спершу вручну (`python sync.py`). *Ціль: дані їдуть із джерела в БД.*
- **5. Розклад.** APScheduler у процесі app, раз на N хв. *Ціль: синхра автоматична.*
- **6. Картинки.** Докачка нових у `static/images`, у БД лише path, `<img>` на сторінці результату. *Ціль: з картинками.*
- **7. Docker.** `Dockerfile` + `compose`, томи для `data/` і `static/images`, bind `0.0.0.0`. *Ціль: один `docker compose up`.*
- **8. Реальна 1С.** Підмінити URL mock на справжній сервіс, звірити формат. *Ціль: бойове джерело.* (коли товариш дасть сервіс)
- **9. Шліфування.** HTML під IE6/QVGA, `є / мало / нема`, конфіг через `.env`, логування, обробка «товар не знайдено».

## Локальна розробка без заліза

Реальний кіоск і коробка не потрібні, щоб писати й тестувати:

- **Джерело 1С** — це `mock_1c.py` (фейковий JSON). Реальну 1С підключаємо аж на етапі 8.
- **Кіоск** — це браузер телефона чи ноута.
- Flask має слухати `0.0.0.0` (не `127.0.0.1`), інакше з телефона не відкриється.
- Дізнатися LAN-IP ноута (`ip a` / `ipconfig`), телефон у тій же Wi-Fi → відкрити `http://<ip-ноута>:5000`.
- У Docker — пробросити порт `-p 5000:5000`.

## Гострі кути та межі

- **HTML тримати примітивним.** Браузер кіоска ~IE6: без `fetch`/AJAX/сучасного CSS. Скан приходить як емуляція клавіатури (DataWedge) у поле вводу + суфікс Enter → проста JS-обробка або автосабміт форми GET-ом.
- **Топологія в продакшені.** Коробка стоїть на LAN кіоска (у магазині), а не там, де ти. VPN — тільки звідти до 1С у провайдера.
- **Безпека.** 1С віддає вузький read-only endpoint, а не весь OData/всю базу назовні.
- **APScheduler у Flask-процесі** — простіше для одного контейнера, але слідкуй за подвійним запуском при ввімкненому reloader-і (`use_reloader=False` у dev). Пізніше за потреби можна винести синхру в окремий процес.
- **Поза скоупом поки що:** автентифікація на кіоску, кілька магазинів/коробок, оновлення прошивки MK500.
